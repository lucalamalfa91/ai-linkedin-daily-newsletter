#1 Latent Space ★ 10/10

Giving Agents Computers — Ivan Burazin, Daytona

May 21, 2026

↳ WHAT'S NEW Daytona, the developer sandbox infrastructure company, is running 850K daily sandbox executions with 74% month-over-month growth — numbers their CEO Ivan Burazin shared on the Latent Space podcast alongside details on their bare metal sandbox architecture and reinforcement learning evaluation infrastructure. ↳ THE REAL STORY The core product is isolated compute environments that agents can actually use — spinning up sandboxes fast enough that an LLM agent can run code, get output, and iterate without the latency killing the loop. Bare metal (rather than nested virtualization) matters here: you get real CPU performance and filesystem access, which is the difference between a sandbox that feels like a dev machine and one that feels like a container inside a container inside a VM. The 850K daily runs figure suggests this is no longer a niche use case — agent scaffolding that needs real compute is becoming a standard architectural layer, not an experiment. The RL evals angle is notable: Daytona is positioning sandboxes as the environment layer for training and evaluating agents, not just serving them.

Claude's take

74% MoM growth is a striking number, but the denominator matters — it's easy to sustain that rate when you're coming off a small base, and the podcast format doesn't surface churn or retention data. The more honest question for anyone evaluating Daytona as infrastructure: what happens to your agent pipeline when the sandbox cold-start adds 2-3 seconds to every tool call? Bare metal helps throughput but doesn't eliminate spin-up latency, and at 850K daily runs, even small per-sandbox overhead compounds fast. Worth pressure-testing before committing your agent architecture to any single sandbox provider.

#2 Latent Space ★ 9/10

All Model Labs are now Agent Labs

May 23, 2026

↳ WHAT'S NEW Every major model lab — OpenAI, Anthropic, Google DeepMind, Meta — has now publicly repositioned around agents, not models. The Latent Space framing captures a structural shift: the product is no longer a model you call, it's a system that acts. ↳ THE REAL STORY For developers, this changes what you're actually integrating against. A model API has a contract: input, output, latency, cost per token. An agent product has a much fuzzier contract: goals, tool access, memory, retry logic, and unpredictable multi-step costs. The labs are moving the abstraction layer up — which means more capability out of the box, but less control over what's happening underneath. If you're building your own orchestration layer today, you're now competing with the vendor's own stack.

Claude's take

↳ WORTH WATCHING When every lab calls itself an agent lab, the word 'agent' stops meaning anything precise. The open question is whether these are genuinely autonomous systems or just multi-step pipelines with better marketing. The catch for builders: vendor lock-in gets significantly stickier when the lab controls not just the model weights but the memory, tool-calling logic, and task loop. Switching costs are about to get much harder to calculate.

#3 Latent Space ★ 9/10

Railway: The Agent-Native Cloud — Jake Cooper

May 20, 2026

↳ WHAT'S NEW Railway is positioning itself as the default deployment layer for AI coding agents — 3M users, 100K signups per week, and reportedly $200K+ in spend coming from coding agents alone. The company runs its own metal (not AWS/GCP resold), which is the infrastructure bet underneath the product story. ↳ THE REAL STORY The signal here is the $200K+ coding agent spend figure. Agents don't just need a place to run code — they need a platform that can handle ephemeral, high-frequency deploy cycles without human approval gates. Railway's pitch is that it was already structured this way (fast deploys, simple config, no Kubernetes ceremony), which makes it accidentally well-suited for agent workflows. Jake Cooper's framing of 'the death of PRs' is the concrete claim: if an agent can deploy directly to a staging or production environment without a human reviewing a pull request, the bottleneck in the software loop moves from deployment to evaluation. Own-metal matters here too — margin control lets Railway price aggressively for the high-volume, low-touch workloads agents generate, which look nothing like a human developer's deploy cadence.

Claude's take

The 'agent-native' label is doing a lot of work right now and Railway isn't the only one wearing it. The honest question is whether Railway's moat is the product or just timing — they happened to be simple and fast before agents needed simple and fast. The own-metal bet is real differentiation on cost structure, but it's also a capital and operational risk that AWS-resellers don't carry. And 100K signups per week is a vanity number until you see what percentage are agents versus humans, and what the retention curve looks like when a coding agent project gets abandoned or the startup behind it folds. Agent-generated workloads could be spiky and churny in ways that make unit economics hard to read from the outside.

#4 Simon Willison ★ 8/10

Datasette Agent

May 21, 2026

↳ WHAT'S NEW Simon Willison released Datasette Agent — an AI assistant built into Datasette that combines his LLM Python library with the Datasette data exploration tool after three years of parallel development. ↳ THE REAL STORY For developers already using Datasette to publish and explore SQLite databases, this means querying data in natural language without leaving the interface. The integration is built on the LLM library, so it inherits LLM's model-agnostic design — you can point it at OpenAI, Anthropic, local models, or anything else LLM supports. The practical upside: analysts who know what they want but not how to write the SQL can now get answers directly, while developers get an extensible base to build custom data agents on top of existing Datasette deployments.

Claude's take

The extensibility claim is worth scrutinizing. 'Extensible AI assistant' often means the happy path works well and everything else requires reading source code. Datasette's plugin ecosystem is mature, but LLM-powered agents that touch arbitrary database schemas have a reliability ceiling — schema complexity, ambiguous column names, and multi-table joins are where natural language to SQL still breaks down regularly. Worth watching whether the agent handles schema introspection gracefully or just passes raw table names to the model and hopes for the best.

#5 Google ★ 8/10

Gemini 3.5 Flash

May 19, 2026

↳ WHAT'S NEW Gemini 3.5 Flash is available via Google's API, with llm-gemini 0.32 adding the gemini-3.5-flash model identifier for CLI and Python access. ↳ THE REAL STORY The excerpt here is thin — what we have is a plugin release confirming the model exists and is callable, not a full technical breakdown. For developers already using Simon Willison's llm toolchain, dropping in gemini-3.5-flash is a one-line swap. The Flash line has historically been Google's price-performance play: faster and cheaper than Pro-tier models, optimized for high-volume inference tasks like classification, summarization pipelines, and agentic loops where you're paying per million tokens at scale.

Claude's take

The source material is a plugin changelog, not Google's actual model card — so benchmark numbers, context window size, pricing, and capability deltas versus Gemini 2.0 Flash are all absent here. Before routing production traffic to it, check the official API docs for rate limits and whether multimodal inputs are supported at parity with the previous Flash generation. The version jump from 2.0 to 3.5 is also non-trivial naming — it skips an entire major version, which either signals a significant internal leap or a marketing reset. Worth knowing which before building assumptions about capability ceilings into your architecture.
