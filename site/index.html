#1 Latent Space ★ 10/10

Giving Agents Computers — Ivan Burazin, Daytona

May 21, 2026

↳ WHAT'S NEW Daytona, a sandbox infrastructure company, is running 850K daily agent executions and reporting 74% month-over-month growth. CEO Ivan Burazin walked through their bare metal sandbox architecture and how they're positioning the product as "agent cloud" — compute purpose-built for AI agents rather than humans. ↳ THE REAL STORY The core bet here is that existing cloud infrastructure (VMs, containers spun up for human developers) is the wrong abstraction for agents. Agents need sandboxes that spin up in milliseconds, run untrusted code safely, and die cleanly — thousands of times per day per customer. Daytona's bare metal approach cuts the virtualization overhead that makes standard cloud containers too slow for tight agent loops. The RL evals angle is telling: they're not just running coding agents, they're running the evaluation harnesses that train them, which means the workload profile is high-parallelism, short-lived, and latency-sensitive in ways that general-purpose cloud isn't optimized for.

Claude's take

74% MoM growth sounds dramatic until you ask what the base was. At early-stage infrastructure companies, that number can mean going from 12 to 21 customers. The 850K daily runs figure is more concrete and harder to fake — but the real question is whether bare metal sandboxes are a durable moat or a temporary gap that AWS, Modal, or E2B closes with a product update. Sandbox infrastructure is becoming crowded fast: Modal, Fly.io, E2B, and now every major cloud provider has an eye on agent compute. Daytona needs to win on either price or cold-start latency before one of those players decides this market is worth a dedicated push.

#2 Latent Space ★ 10/10

Railway: The Agent-Native Cloud — Jake Cooper

May 20, 2026

↳ WHAT'S NEW Railway is positioning itself as the default deployment layer for AI coding agents — 3M users, 100K signups per week, and reportedly $200K+ in monthly spend coming from coding agents deploying and managing infrastructure autonomously. The company runs its own metal in data centers rather than reselling AWS or GCP capacity. ↳ THE REAL STORY The $200K+ coding agent spend figure is the number worth sitting with. It means agents — Cursor, Claude, Codex, whatever is in your loop — are already provisioning real infrastructure, not just writing code that humans then ship. Railway is betting that the deployment target for agentic workflows needs to be designed around non-human operators: fast API-driven provisioning, no UI friction, predictable pricing that an agent can reason about. Running own-metal gives them margin control and latency consistency that resellers can't match when you're doing thousands of small, short-lived deployments. The death-of-PRs framing is the real thesis: if agents commit, test, and deploy autonomously, the human review gate disappears and the cloud layer needs to keep up with that cadence.

Claude's take

The own-metal bet is interesting but cuts both ways. It gives Railway cost and performance advantages at scale, but it also means they're on the hook for capacity planning in a market where agent-driven demand spikes are genuinely hard to forecast. If coding agent usage doubles in a quarter — plausible given current trajectory — do they have the runway to rack servers fast enough? The 100K signups per week number also needs a denominator: conversion to paid, retention past the first deploy, and whether agent-generated spend is sticky or just noise from people kicking tires on vibe-coded side projects. The agent-native narrative is compelling, but the infrastructure business underneath it is still a infrastructure business.

#3 Simon Willison ★ 9/10

Datasette Agent

May 21, 2026

↳ WHAT'S NEW Simon Willison shipped Datasette Agent — a first release of an AI assistant built directly into Datasette, combining his LLM Python library with the Datasette data exploration tool after three years of parallel development. ↳ THE REAL STORY This is the practical payoff of building LLM as a general-purpose library rather than a one-off integration. Datasette Agent means you can point a model at a SQLite database and query it in natural language without wiring up your own tool layer — the scaffolding (schema introspection, query execution, result formatting) is already handled. For developers who use Datasette to share or explore datasets, this removes the step of building a custom SQL-generation wrapper every time. The extensible architecture also means the agent can be adapted with plugins, which fits Datasette's existing ecosystem.

Claude's take

The excerpt cuts off before the technical details, so the actual model support, tool definitions, and safety guardrails are unknown from this alone. The bigger open question: SQL generation agents have a well-documented failure mode on ambiguous schemas or multi-table joins — and SQLite's permissive type system makes silent errors easy to miss. Worth checking whether Datasette Agent exposes the generated SQL to the user before execution, or runs it blind. That single design choice separates a useful tool from a liability when the underlying data actually matters.

#4 Latent Space ★ 9/10

[AINews] All Model Labs are now Agent Labs

May 23, 2026

↳ WHAT'S NEW Latent Space flags a structural shift across the industry: model labs — OpenAI, Anthropic, Google DeepMind, and others — are repositioning themselves as agent labs. The framing isn't about a single product drop but a convergence of strategic messaging and product direction happening simultaneously across the board. ↳ THE REAL STORY The practical implication is that the competitive surface has moved. Labs are no longer racing primarily on benchmark scores or context windows — they're racing on who owns the agentic runtime: the orchestration layer, the tool-use protocols, the memory systems, the eval frameworks for multi-step tasks. For developers building today, this means the APIs you're integrating against are going to keep shifting underneath you as labs bolt on agent infrastructure. Picking a stack right now means betting on whose agent abstraction survives — and most of them are still half-baked.

Claude's take

The catch is that 'agent lab' is doing a lot of marketing work right now with very little standardization behind it. There's no shared definition of what an agent is, no common eval benchmark the industry agrees on, and the tooling — memory, planning, error recovery — is still genuinely unsolved at production scale. When every lab calls itself an agent lab in the same quarter, it's worth asking whether the category is maturing or whether everyone is just chasing the same pitch deck. The companies that actually deliver reliable multi-step task completion will look very different from the ones that shipped a system prompt wrapper and called it an agent.

#5 Simon Willison ★ 8/10

datasette-agent 0.1a4

May 24, 2026

↳ WHAT'S NEW datasette-agent 0.1a4 ships, built on top of the new makeJumpSections() JavaScript plugin hook introduced in a recent Datasette alpha. The release wires an AI agent directly into the Datasette UI, letting it navigate and query SQLite databases through the existing plugin architecture. ↳ THE REAL STORY For developers already running Datasette as a data exploration layer, this means an agent can now traverse database structure, run SQL, and surface results without any custom scaffolding — the plugin hook handles UI integration. The practical upside: you get an LLM-assisted query interface on top of any SQLite file you're already serving through Datasette, with no separate backend to maintain. The agent operates within the constraints of Datasette's permission and plugin model, which keeps it from being a free-floating SQL executor.

Claude's take

Still an early alpha (0.1a4), so the API surface will move. The more interesting question is how the agent handles Datasette's read-only vs. writable instance modes — if it can only read, it's a solid exploration tool; if write access is possible through the agent, that's a different risk profile entirely. Worth watching whether Willison documents the permission boundaries explicitly before this stabilizes.
