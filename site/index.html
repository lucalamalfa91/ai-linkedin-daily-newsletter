#1 Latent Space ★ 10/10

Giving Agents Computers — Ivan Burazin, Daytona

May 21, 2026

↳ WHAT'S NEW Daytona, the developer sandbox infrastructure company, is running 850K daily sandbox executions and reporting 74% month-over-month growth. CEO Ivan Burazin joined Latent Space to break down how they're building bare-metal sandbox environments specifically for AI agents that need to run code, browse the web, or manipulate files — the "computer use" layer of the agent stack. ↳ THE REAL STORY The core problem Daytona is solving is real and underappreciated: when an agent needs to execute code or interact with a filesystem, you need an isolated, fast-booting environment that doesn't cost a fortune to keep warm and doesn't blow up your production infra if the agent does something stupid. Bare-metal sandboxes mean lower virtualization overhead compared to nested containers or full VMs — faster cold starts, more predictable I/O. At 850K daily runs, they're also sitting on a dataset of agent execution patterns that feeds directly into their RL eval work, which is the more interesting long-term play: using real execution traces to evaluate and improve agent behavior rather than relying on synthetic benchmarks.

Claude's take

↳ WORTH WATCHING 74% MoM growth is a striking number, but the sandbox infrastructure space is filling up fast — E2B, Modal, Morph, and now every major cloud provider has something in this lane. The defensible moat here isn't the sandbox itself; it's whether the RL eval layer built on top of execution data becomes genuinely proprietary signal. If Daytona stays primarily an infrastructure vendor, margin pressure from AWS and GCP is inevitable. The open question is whether "agent cloud" becomes a real category with its own primitives — or just gets absorbed into existing compute platforms as a feature.

#2 Latent Space ★ 10/10

Railway: The Agent-Native Cloud — Jake Cooper

May 20, 2026

↳ WHAT'S NEW Railway is positioning itself as infrastructure built specifically for AI coding agents — not just a PaaS that tolerates them. The platform reports 3M users, 100K signups per week, runs on owned metal (not AWS/GCP resale), and is seeing individual customers spend $200K+ on coding agent workloads. ↳ THE REAL STORY The $200K agent spend figure is the one worth sitting with. That's not a team of developers deploying apps — that's autonomous agents spinning up environments, running evals, deploying builds, and tearing things down at a cadence no human workflow produces. Railway's bet is that the deploy-test-iterate loop agents run is fundamentally different from what Heroku-style PaaS was designed for: higher frequency, shorter-lived environments, less human oversight at each step. Owning the metal matters here — margin and latency control become competitive advantages when your customer is a loop running hundreds of deploys a day, not an engineer clicking a button once. The 'death of PRs' framing suggests the platform is optimizing for merge-less, agent-driven continuous deployment — where the human review gate is removed or moved much earlier in the process.

Claude's take

The 100K signups per week number sounds impressive until you ask what the activation and retention curve looks like — Railway has historically attracted developers who kick the tires and leave. The more pointed question is whether 'agent-native' is a durable architectural differentiator or a positioning layer on top of standard container orchestration. Any cloud provider can add an agent-friendly API surface. What Railway actually has is a cost structure from owned metal and a developer-experience reputation — neither of which is unique to agent workloads. If the real moat is just 'we're cheaper and faster for ephemeral environments,' that's a thin one once AWS decides to compete on that dimension directly.

#3 Latent Space ★ 9/10

[AINews] All Model Labs are now Agent Labs

May 23, 2026

↳ WHAT'S NEW Latent Space flags a structural shift across the AI industry: model labs — OpenAI, Anthropic, Google DeepMind, and others — are repositioning themselves as agent labs. The framing isn't about a single product drop; it's about where every major lab is now pointing its roadmap. ↳ THE REAL STORY The competitive surface has moved. Labs that built their identity around foundation model benchmarks are now racing to own the agentic layer — task execution, tool use, multi-step reasoning, memory, and orchestration. For developers, this means the commodity pressure that hit raw text generation is now creeping up the stack. If OpenAI ships a native agent runtime, the third-party orchestration framework you're building on top of GPT-4 gets a lot more exposed. The practical question for anyone building today: are you constructing on infrastructure that a lab considers its next core product?

Claude's take

The catch is that 'agent lab' is doing a lot of work as a label right now. Labs calling themselves agent-focused doesn't mean their agents are reliable enough to replace the orchestration glue developers have already built and battle-tested. Benchmark performance on agentic tasks still diverges sharply from production behavior — tool-call failure rates, context management under long horizons, and cost per completed task remain unsolved in ways that a rebrand doesn't fix. Watch whether the labs ship agent evals alongside agent products, or just ship the marketing.

#4 Simon Willison ★ 9/10

Datasette Agent

May 21, 2026

↳ WHAT'S NEW Simon Willison shipped Datasette Agent — an AI assistant built into Datasette that combines his LLM Python library with the Datasette data exploration tool after three years of parallel development. ↳ THE REAL STORY This is the practical payoff of building LLM as a standalone library rather than coupling it to a single product. Datasette Agent can query SQLite databases through natural language, with the LLM library handling model routing underneath — meaning you can swap backends (OpenAI, Anthropic, local models) without rewriting the agent logic. For developers already running Datasette for data exploration, this adds a conversational layer on top of existing datasets without standing up a separate service.

Claude's take

The extensibility claim is the part worth pressure-testing. 'Extensible AI assistant' usually means plugin hooks exist on paper but the happy path is one specific model provider. Worth watching whether the agent handles multi-table joins and ambiguous schema gracefully, or whether it degrades into hallucinated SQL the moment your database gets non-trivial — which is exactly where most natural-language-to-SQL tools quietly fall apart.

#5 Simon Willison ★ 8/10

datasette-agent 0.1a4

May 24, 2026

↳ WHAT'S NEW datasette-agent 0.1a4 ships, built on top of the new makeJumpSections() JavaScript plugin hook introduced in a recent Datasette alpha. The release wires an LLM agent directly into the Datasette UI, letting users query SQLite databases through a conversational interface without leaving the browser. ↳ THE REAL STORY For developers already running Datasette as a local or internal data exploration tool, this means you can ask natural-language questions against any loaded SQLite file and get SQL generated, executed, and rendered inline — no separate chat window, no copy-pasting schemas. The makeJumpSections() hook is what lets the agent inject its output as a first-class UI panel rather than a bolted-on overlay. Early alpha status means the API surface will shift, but the core loop — describe what you want, get a query, see results — is functional enough to replace a lot of manual SQL drafting for exploratory work.

Claude's take

The honest question here is how well the agent handles schema ambiguity at scale. Datasette is often pointed at messy, undocumented SQLite exports — the exact databases where column names are cryptic and relationships are implicit. LLM-generated SQL tends to hallucinate joins or misread types when there's no clean schema to anchor on. Until there's evidence the agent does retrieval-augmented schema introspection rather than just dumping the full schema into context, treat it as reliable for small, well-structured databases and unreliable for anything else. Also worth noting: 0.1a4 is pre-release software from a solo maintainer — production dependency is premature.
