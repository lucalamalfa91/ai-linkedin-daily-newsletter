#1 Cursor ★ 10/10

Cursor Automations in Agents Window

May 20, 2026

↳ WHAT'S NEW Cursor Automations moved into the Agents Window — previously only accessible at cursor.com/automations, they now live alongside agents in the same interface. Two new automation types ship with this: multi-repo automations (agents reasoning across multiple codebases simultaneously) and no-repo automations (agents that monitor external tools like Slack or data warehouses and act on signals without any attached repository). ↳ THE REAL STORY The consolidation is mostly a UX win — one less context switch between managing automations and managing agents. The more substantive change is the no-repo automation type. It lets you wire an agent to a non-code signal source — a Slack message, a warehouse query result, a metric threshold — and have it act without needing a codebase as the entry point. That's a meaningful expansion of where Cursor agents can sit in a workflow: not just "triggered by a PR" but "triggered by anything with an API." Multi-repo support matters for monorepo-adjacent setups where a change in a shared library needs to be reasoned about in the context of downstream consumers.

Claude's take

↳ WORTH WATCHING The no-repo automation path is the one to stress-test carefully. Agents acting on external signals without a code anchor — monitoring Slack, querying warehouses — is where scope creep and unintended side effects get hard to audit. Cursor hasn't published much on the permission model or action boundaries for these automations. Before wiring one to anything production-adjacent, the open question is: what does the blast radius look like if the agent misreads a signal, and is there a meaningful approval gate before it acts?

#2 Amazon Q ★ 9/10

Amazon Q Developer agentic capabilities on SWE-Bench

May 23, 2026

↳ WHAT'S NEW Amazon Q Developer claimed the top spot on both SWE-Bench and SWE-Bench Lite leaderboards — the standard benchmarks for evaluating AI agents on real GitHub issues across open-source Python repos. ↳ THE REAL STORY SWE-Bench measures whether an agent can actually resolve filed GitHub issues end-to-end: read the repo, locate the bug, write a fix, pass the tests. Topping both the full and Lite variants suggests Q Developer's agentic loop — planning, file editing, test execution — is genuinely competitive on multi-step tasks, not just autocomplete. For teams already on AWS, this is a concrete reason to revisit Q Developer for PR review automation and test generation rather than treating it as a second-tier Copilot alternative.

Claude's take

↳ WORTH WATCHING SWE-Bench scores are notoriously sensitive to evaluation methodology: which scaffold, how many retries, whether the agent sees the test suite. Amazon hasn't published a technical report detailing their setup, which makes the number hard to verify or reproduce. The leaderboard also shifts fast — a top score today can be third place in six weeks. Before routing real engineering workflows through Q Developer based on this claim, it's worth waiting for an independent replication or at least a detailed methodology disclosure from Amazon.

#4 OpenAI Codex ★ 8/10

Secure MCP Tunnel for enterprise customers

May 19, 2026

↳ WHAT'S NEW OpenAI's Secure MCP Tunnel lets enterprise customers connect private or on-premises MCP servers to ChatGPT web, Codex, Responses API, and AgentKit — via a customer-hosted tunnel client, no direct internet exposure required. ↳ THE REAL STORY The blocker for most enterprise MCP adoption has been straightforward: internal tools, databases, and APIs sit behind firewalls that OpenAI's infrastructure can't reach. This tunnel client flips the connection direction — your infrastructure initiates outbound, OpenAI's products consume it. In practice, that means a Codex agent can now call an internal Jira instance, a private code registry, or an on-prem data warehouse without punching holes in the firewall or routing sensitive data through a public endpoint. The integration surface for agentic workflows just got significantly wider for regulated industries.

Claude's take

↳ WORTH WATCHING The tunnel client is customer-hosted — which means your team owns the operational burden: deployment, uptime, patching, and monitoring. If the tunnel goes down, every connected OpenAI product loses access to those MCP servers silently or noisily depending on how well you've instrumented it. There's also a trust question worth sitting with: traffic still transits OpenAI's infrastructure on the other end of that tunnel. For teams in heavily regulated environments (finance, healthcare, government), 'no direct internet exposure' is not the same as 'data never leaves our perimeter' — and that distinction will matter when the compliance team asks.

#5 Continue.dev ★ 6/10

Filter session history by workspace directory

March 27, 2025

↳ WHAT'S NEW Continue.dev added workspace-scoped filtering to session history — chat logs can now be viewed per workspace directory instead of as one global list. ↳ THE REAL STORY If you run Continue across several repos simultaneously, the session list was previously a flat pile of conversations with no project context. Now you can open a workspace and see only the chats that happened there. Practically: faster retrieval of that debugging thread from last Tuesday without scrolling past unrelated sessions from three other codebases. Small UX change, real friction removed for anyone juggling more than two active projects.

Claude's take

The feature is only as useful as your workspace hygiene. If you habitually open files ad-hoc outside a defined workspace root, or work in monorepos where multiple logical projects share one folder, the filter won't give you the granularity you actually want. Worth checking whether Continue scopes by the exact root path or does something smarter — because a monorepo with five services all under one directory will still show you everything.
