#3 Latent Space ★ 9/10

The Age of Async Agents — Cognition's Walden Yan & OpenInspect's Cole Murray

May 28, 2026

↳ WHAT'S NEW Cognition's Walden Yan and OpenInspect's Cole Murray joined Latent Space to break down where Devin actually stands: 80% of commits in some workflows are now Devin-authored, with spec-to-PR pipelines running inside full VMs, persistent agent memory across sessions, and product managers shipping code without engineering handoffs. ↳ THE REAL STORY The 80% commit figure is the number worth sitting with. If accurate in even a narrow set of codebases, it reframes Devin from 'AI pair programmer' to 'primary contributor with human review.' The spec-to-PR workflow is the concrete mechanism: a PM writes a spec, Devin spins up an isolated VM, runs the full dev loop — write, test, iterate — and opens a pull request. The human touches the diff, not the keyboard. Persistent memory means Devin retains context about your repo conventions and past decisions across runs, which is what previously made autonomous agents fall apart on anything beyond toy tasks.

Claude's take

The 80% figure almost certainly reflects cherry-picked workflows — greenfield features, well-scoped tickets, codebases Devin has already indexed. The harder question is what the other 20% looks like: is it the easy stuff humans still prefer to own, or is it the hard stuff Devin still can't touch? Async agents that fail silently on complex tasks and return a plausible-looking but broken PR are a different risk profile than a junior dev who asks clarifying questions. Until there's public data on failure modes and rollback rates, the PM-ships-code narrative is a best-case framing.

#5 Datasette

SQL write queries and stored queries in Datasette 1.0a31

May 29, 2026

↳ WHAT'S NEW Datasette 1.0a31 adds two features to the alpha track: permission-gated SQL write query execution directly from the UI, and the ability to save and store named queries against a database. ↳ THE REAL STORY Write queries in the UI close a long-standing gap — until now Datasette was a read-only SQL interface, meaning any mutation required dropping to the CLI or a separate tool. For teams using Datasette as a lightweight internal data tool, this means editors can run INSERT/UPDATE/DELETE statements without touching the underlying server. Stored queries are the more quietly useful addition: named, reusable SQL snippets that live in the database and can be shared via URL, which removes the copy-paste workflow that most Datasette users have been living with.

Claude's take

The permission model is the thing to watch closely. Write access in a browser-facing SQL interface is a meaningful attack surface — one misconfigured permission and you have arbitrary writes on your production SQLite file. Datasette's permission system has historically been opt-in and easy to misconfigure, especially in self-hosted setups without a reverse proxy. Before using write queries in anything beyond a local dev environment, it's worth auditing exactly which actors can reach the `execute` permission and whether the alpha-track API for that will stay stable before 1.0 final.
