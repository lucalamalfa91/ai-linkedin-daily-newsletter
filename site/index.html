#1 Cursor ★ 9/10

Composer 2.5 Model Release

May 18, 2026

Cursor released Composer 2.5, a new model for its agent/composer workflow with two pricing tiers: Standard at $0.50/M input and $2.50/M output, and Fast (default) at $3.00/M input and $15.00/M output — a 6x cost differential between tiers. The model targets long-running agentic tasks specifically, with claimed improvements in sustained multi-step reasoning and complex instruction following over Composer 2. Mechanistically, this appears to be a separate model checkpoint optimized for tool-use and instruction chaining rather than a fine-tune of an existing public model. Cursor is doubling included usage for the first week, so teams can benchmark 2.5 against 2 on their actual workloads at no extra cost before committing to the Fast tier.

Claude's take

For a developer running a large refactor — say, renaming an interface across 80 files with cascading type fixes — the difference between Standard and Fast tier matters: if Fast has meaningfully lower latency per tool call, multi-step chains complete before context drift sets in, but at 6x the token cost that adds up fast on long sessions. The non-obvious implication is that the two-tier pricing structure creates a new optimization problem: teams will need to profile which task classes actually benefit from Fast, since paying $15/M output for tasks that Standard handles equally well is pure waste. This release directly addresses a known Cursor pain point where Composer 2 would lose coherence or start hallucinating tool calls midway through tasks involving more than ~10 sequential steps. The honest gotcha: 'improved instruction following' and 'better sustained work' are unverified marketing claims until you test on your own codebase — and with no published benchmark data or architectural details, there's no way to predict whether your specific failure modes with Composer 2 are actually fixed.

#2 Amazon Q ★ 8/10

Amazon Q Developer agentic capabilities on SWE-Bench

May 22, 2026

Amazon Q Developer's agentic capabilities claimed top positions on both SWE-Bench and SWE-Bench Lite leaderboards, which evaluate AI systems on real GitHub issues requiring multi-file code changes across Python repositories. SWE-Bench measures an agent's ability to autonomously resolve issues by editing codebases — the benchmark runs agents against 300 (Lite) or 2,294 (full) real-world issues and scores on whether the resulting patch passes the associated test suite. The announcement covers multi-step workflows including unit test generation, documentation, and code review automation, though specific resolution-rate percentages are not disclosed in this release. Developers can now point to an externally validated benchmark result when evaluating Q Developer against alternatives like Devin, SWE-agent, or Copilot Workspace for autonomous issue resolution.

Claude's take

The most concrete use case is dropping Q Developer into a CI triage workflow: when a failing test or GitHub issue lands, an agent that scores well on SWE-Bench should be able to propose a patch across multiple files without hand-holding — measurably reducing the back-and-forth on routine bug fixes in mid-sized Python or Java repos. The non-obvious implication is that SWE-Bench performance correlates heavily with how well an agent handles repo navigation and test-driven feedback loops, so a top score suggests Q's underlying tool-use and context management is meaningfully better than prior versions, not just its code generation. This matters now because the agentic coding space is consolidating fast — Devin, GitHub Copilot Workspace, and Cursor are all targeting the same 'autonomous PR' workflow, and benchmark positioning is becoming a real procurement signal for engineering orgs evaluating these tools. The honest gotcha: SWE-Bench is Python-centric and the issues are drawn from a fixed set of open-source repos, so top-of-leaderboard performance may not transfer cleanly to your internal Java monorepo or proprietary framework — treat the benchmark as a signal, not a guarantee, and run your own evals on representative tasks before committing to the workflow.

#3 Continue.dev ★ 6/10

Filter session history by workspace directory

March 27, 2025

Continue.dev added workspace-directory-scoped filtering to the session history panel. When selected, the filter restricts the history list to sessions that were initiated within a specific workspace folder, matching on the root directory path stored with each session record. No context-window or token changes are involved — this is purely a UI/navigation feature. Developers working across multiple repos can now pull up only the chat history relevant to the project they're currently in, without scrolling through unrelated sessions from other codebases.

Claude's take

If you're context-switching between, say, a frontend monorepo and a backend service daily, this directly cuts the time spent hunting for a session where you debugged a specific API — instead of scanning 50 mixed entries, you filter to the 8 sessions for that repo. The non-obvious implication: session history scoped by workspace becomes a lightweight per-project knowledge base, which means the quality of your session naming and the habit of staying in the right workspace root now has compounding value. This addresses a real pain point as Continue usage scales — global history becomes noise fast once you're running it across 5+ active projects. One gotcha: the filter almost certainly matches on the workspace root path string, so renamed or moved directories will orphan historical sessions from the filter — don't reorganize your project folders and expect old history to follow.

#4 Claude Code Docs ★ 10/10

Claude Code Hooks: Synchronous Middleware for Every Tool Call

May 22, 2026

Claude Code now ships a formal hooks system that injects user-defined shell commands, HTTP endpoints, or LLM prompts at 25+ named lifecycle points in the agentic loop — SessionStart, PreToolUse, PostToolUse, WorktreeCreate, and more. Each hook receives a structured JSON payload on stdin describing the tool call in progress, and can return a permissionDecision that Claude must honor before execution continues — this is synchronous blocking, not an async observer pattern. A two-stage filtering system (tool-name matcher + regex/pattern match against the tool's actual input payload) means hook processes only spawn when both conditions are met, eliminating overhead for non-matching calls. The net result: you can now enforce arbitrary pre/post conditions on every file write, shell command, or sub-agent action without forking Claude's CLI or maintaining a wrapper process.

Claude's take

The concrete scenario where this pays off immediately: in a 300+ file monorepo with Claude doing autonomous refactors, you can attach a PreToolUse hook on Write/Edit tools that runs a schema validator or lint check against the proposed file content before it lands on disk — catching drift that would otherwise require a separate CI cycle to surface. The non-obvious second-order effect is that hooks compose into a lightweight audit trail: PostToolUse hooks logging every tool invocation to a structured sink give you a replay log of exactly what Claude did and in what order, which is genuinely useful for debugging non-deterministic agent runs. This lands now because agentic Claude Code sessions have been a trust gap — developers were either fully delegating or babysitting every step, with no middle layer; hooks are that middle layer. The gotcha: hooks are shell-level, so they run with your local user permissions and any hook that calls an LLM prompt adds latency to every matched tool call — if your matcher is too broad, you can accidentally serialize the entire agentic loop behind an LLM round-trip, which will make long autonomous sessions noticeably slower.

#5 Claude Code Docs ★ 5/10

MCP in Claude Code: Live Tool Access That Kills the Copy-Paste Loop

May 22, 2026

Claude Code now supports Model Context Protocol (MCP) servers as first-class tool providers, letting Claude act directly on external systems — issue trackers, databases, CI pipelines — within a session rather than just reading about them. Two transport modes are supported: local stdio servers spawn as child processes (with CLAUDE_PROJECT_DIR injected into their environment), and remote servers communicate over streamable-HTTP transport, both treated identically to Claude's built-in tools. You configure servers via `claude mcp add` for project-scoped or user-scoped registries, or drop a `.mcp.json` into the repo root for team-shared configs. The practical upshot: Claude can now open a PR, query a database, or trigger a deployment as a native step in a multi-turn session — no copy-pasting tool output back into the prompt.

Claude's take

In a concrete workflow: if you wire a GitHub MCP server to a Claude Code session on a 200-file repo, Claude can read an open issue, locate the relevant files, write the fix, and open the PR in a single prompt chain — the round-trip that previously required 4-5 manual copy-paste steps collapses to one. The non-obvious gotcha the docs bury: stdio servers receive CLAUDE_PROJECT_DIR as an injected env var, not inherited from Claude's own process environment — so any custom server doing project-relative path resolution must explicitly read that variable, or you'll get silent path mismatches that are annoying to debug. This lands at exactly the moment teams are hitting the ceiling of 'Claude as advisor' and pushing toward 'Claude as executor' — MCP is the plumbing that makes agentic pipelines composable without bespoke glue code. The honest trade-off: stdio child processes mean your MCP server's crash or hang directly affects the Claude Code session, and there's no built-in sandboxing — a poorly written custom server with broad filesystem access is a supply-chain risk if you're pulling community MCP servers without auditing them.
