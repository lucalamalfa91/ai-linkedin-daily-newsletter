#1 Latent Space ★ 9/10

Giving Agents Computers — Ivan Burazin, Daytona

May 21, 2026

↳ WHAT'S NEW Daytona, a sandbox infrastructure company, is running 850K daily agent executions and reporting 74% month-over-month growth — numbers their CEO Ivan Burazin shared on the Latent Space podcast. The product: bare metal sandboxes purpose-built for giving AI agents access to computers, filesystems, and code execution environments. ↳ THE REAL STORY The growth signal here is the underlying demand curve for agent infrastructure, not Daytona specifically. As more teams move from "LLM that answers questions" to "LLM that runs code and operates tools," they hit the same wall: cloud VMs are too slow to spin up, too expensive to keep warm, and too risky to share across untrusted agent workloads. Bare metal sandboxes solve the isolation and latency problem — each agent run gets a clean, fast environment without the overhead of full VM provisioning. At 850K daily runs, Daytona is also sitting on a rare dataset: what agents actually do when given a computer, which feeds directly into their RL evals work. That eval layer is the less-discussed part — teams building agents need to know if the agent actually completed the task correctly, not just whether it ran without errors.

Claude's take

↳ WORTH WATCHING The 74% MoM growth number is eye-catching but needs context: growing fast from a small base is very different from sustaining that rate at scale. The more interesting open question is whether bare metal sandbox providers become critical infrastructure or get commoditized by the hyperscalers. AWS, Google, and Azure all have incentives to offer native agent execution environments — and they already own the compute. Daytona's defensible moat, if it has one, is probably the eval and observability layer built on top of those 850K daily runs, not the sandboxes themselves. Watch whether they move up the stack toward agent orchestration or stay as pure infrastructure.

#2 Latent Space ★ 9/10

Railway: The Agent-Native Cloud — Jake Cooper

May 20, 2026

↳ WHAT'S NEW Railway is positioning itself as the default deployment target for AI coding agents — 3M users, 100K signups per week, running on owned metal rather than hyperscaler infrastructure, and reportedly seeing $200K+ in spend from coding agents deploying and iterating on apps autonomously. ↳ THE REAL STORY The interesting number here isn't the user count — it's the agent spend figure. Coding agents like Claude Code, Cursor, and Devin need somewhere to push, run, and test code without a human in the loop approving pull requests. Railway is betting that the PR-based deployment model breaks down when the committer is a bot running at 3am. Their owned-metal setup matters here: lower marginal cost per deploy means they can absorb the high-frequency, low-ceremony deploys that agents generate — dozens of iterations per task rather than one careful human push. If agents are your primary customer, you optimize for API surface and deploy speed, not the GitHub Actions integration your DevOps team spent a week tuning.

Claude's take

The $200K agent spend figure is striking but needs context — is that total platform revenue from agent-initiated deploys, or a single customer anecdote? The 'death of PRs' framing is also doing a lot of work. Most teams using coding agents still gate production deploys behind human review; what Railway is really capturing is the sandbox and staging layer, not production. The open question is whether that wedge — own the agent's test environment — is enough to move up the stack into production workloads, or whether it stays a clever niche while AWS and GCP add their own agent-friendly deploy primitives.

#3 Latent Space ★ 8/10

[AINews] All Model Labs are now Agent Labs

May 23, 2026

↳ WHAT'S NEW The major AI model labs — OpenAI, Anthropic, Google DeepMind, and others — are collectively pivoting their public identity and product roadmaps from "model providers" to "agent platforms." The shift isn't a single announcement but a pattern visible across recent releases, hiring signals, and positioning language. ↳ THE REAL STORY This matters for builders because it changes what these companies optimize for. A model lab ships better benchmarks and API capabilities. An agent lab ships orchestration primitives, memory systems, tool-use reliability, and long-horizon task infrastructure. If the framing sticks, expect R&D and product investment to flow toward things like persistent state, multi-step reliability, and computer-use interfaces — not just raw model quality. Developers who have been stitching together their own agent scaffolding on top of raw APIs may find that scaffolding increasingly commoditized or absorbed into the platforms themselves.

Claude's take

↳ WORTH WATCHING The catch is that "agent lab" is still mostly a positioning move, not a capability reality. Reliability on multi-step tasks remains the unsolved problem — most production agent failures happen not because the model is too weak but because error propagation across steps is uncontrolled. Labs calling themselves agent platforms before that problem is solved risks setting expectations that current systems cannot meet. Watch whether the labs ship better evals and reliability tooling alongside the branding, or whether this is just a narrative layer on top of the same underlying models.

Datasette Agent

May 21, 2026

↳ WHAT'S NEW Datasette Agent is the first release of an AI assistant built into Datasette, combining Simon Willison's LLM Python library with the Datasette data exploration tool after three years of parallel development. ↳ THE REAL STORY For developers already using Datasette to publish and explore SQLite databases, this means natural language querying and data interaction without leaving the tool. The LLM library handles model abstraction, so the agent isn't locked to a single provider — any backend supported by LLM (OpenAI, Anthropic, local models via Ollama, etc.) should work. The practical upside: analysts who know what they want from a dataset but don't want to hand-write SQL can now iterate faster, while developers get an extensible foundation they can hook into via plugins.

Datasette's plugin ecosystem is genuinely good, but 'extensible AI assistant' often means the core behavior is narrow and the extension points are underspecified at launch. Worth watching whether the agent can handle multi-step reasoning across joined tables or whether it's essentially a SQL autocomplete wrapper — the difference matters enormously for real analytical workloads. Also: the excerpt cuts off, so the full scope of what 'first release' actually ships is still unclear from the announcement alone.

#5 Simon Willison ★ 7/10

datasette-agent 0.1a4

May 24, 2026

↳ WHAT'S NEW datasette-agent 0.1a4 ships, built on top of the new makeJumpSections() JavaScript plugin hook introduced in a recent Datasette alpha. The release wires an AI agent directly into the Datasette UI, letting it navigate and query SQLite databases through the existing plugin architecture. ↳ THE REAL STORY The interesting part here is the integration pattern: rather than bolting an LLM onto a separate endpoint, datasette-agent hooks into Datasette's own JavaScript plugin system. That means the agent can surface results inline — jumping to relevant table sections, running SQL, and presenting output — without requiring a separate backend service. For anyone already running Datasette as a local data exploration tool, this is a low-friction way to add natural-language querying on top of existing SQLite files. No new infrastructure, just a plugin install.

Claude's take

This is still alpha (0.1a4), and the excerpt cuts off before the full changelog — so the actual scope of what the agent can and cannot do remains unclear from the release notes alone. The deeper question is tool reliability: LLM-generated SQL against arbitrary SQLite schemas fails in predictable ways on joins, ambiguous column names, and large result sets. Whether datasette-agent adds guardrails around those failure modes, or just passes raw model output to the query engine, will determine whether this is genuinely useful or a demo that breaks on real-world messy data.
